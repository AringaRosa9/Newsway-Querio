## 1. 产品概述

### 1.1 产品定位

面向专业用户（媒体人、分析师、研究者）的 AI 驱动新闻搜索引擎。

**一句话描述**：用 AI 让新闻搜索"搜得准、看得快、追得住"。

### 1.2 目标用户

| 用户角色 | 核心诉求 | 使用场景 |
|---|---|---|
| 媒体从业者 | 快速获取某事件的全貌和最新进展 | 写稿前调研、追踪持续性事件 |
| 行业分析师 | 获取特定行业/公司的资讯并做趋势研判 | 行业研报撰写、竞品监控 |
| 学术研究者 | 检索特定主题的新闻语料并做内容分析 | 舆情研究、案例收集 |
| 企业公关 | 监控品牌/竞品的媒体曝光 | 舆情预警、危机响应 |

### 1.3 与竞品的差异化

| 维度 | 传统新闻搜索（Google News / 百度资讯） | 本产品 |
|---|---|---|
| 输入方式 | 关键词 | 自然语言提问 |
| 结果形态 | 链接列表 | AI 综合摘要 + 溯源链接 |
| 信息组织 | 按篇排列 | 按事件聚合，生成时间线 |
| 持续跟踪 | 需反复手动搜索 | 订阅事件，自动推送新进展 |
| 搜索理解 | 字面匹配为主 | 意图理解 + 语义检索 |

## 3. 功能需求

### 3.1 P0 · 智能搜索（核心）

#### 3.1.1 自然语言查询

**描述**：用户使用日常语言输入搜索意图，系统自动进行意图识别和 Query 改写。

**功能点**：

- 支持自然语言输入，如"最近一周哪些科技公司裁员了？"
- 系统自动做 Query 理解，提取关键实体、时间范围、搜索意图
- 对模糊查询进行 Query 改写和扩展，提升召回率
- 支持中英文混合输入

**验收标准**：

- 自然语言 query 的搜索结果相关性不低于精确关键词搜索
- Query 理解的意图识别准确率 ≥ 90%
- 响应时间 ≤ 3 秒（含 AI 摘要生成）

#### 3.1.2 混合检索

**描述**：结合传统关键词检索和语义向量检索，提升搜索的准确性和召回率。

**功能点**：

- BM25 关键词检索：精确匹配用户查询中的关键词
- Embedding 语义检索：基于向量相似度匹配语义相关的内容
- 双路召回结果融合：加权合并两路结果，去重后送入排序层

**验收标准**：

- 混合检索的 Recall@50 ≥ 0.85
- 相比纯关键词检索，相关结果召回提升 ≥ 20%

#### 3.1.3 AI Re-Ranking

**描述**：对召回结果进行多维度智能排序。

**排序因子**：

| 因子 | 权重（参考） | 说明 |
|---|---|---|
| 相关性 | 40% | Query 与文档的语义匹配度 |
| 时效性 | 25% | 发布时间越近权重越高，有衰减曲线 |
| 权威性 | 20% | 来源媒体的公信力评级 |
| 多样性 | 15% | 避免结果来源过于集中 |

**验收标准**：

- NDCG@10 ≥ 0.75
- 用户首次点击平均位置 ≤ 第 3 条

#### 3.1.4 AI 摘要回答

**描述**：基于搜索结果 Top-N 文档，生成综合性的摘要回答，并附带原文引用。

**功能点**：

- 对 Top-10 相关文档进行 AI 综合摘要
- 摘要中标注信息来源，用户可点击跳转原文
- 多角度呈现：事实概述、各方观点、发展趋势
- 标注摘要生成时间和信息来源数量

**验收标准**：

- 摘要事实准确率 ≥ 95%（无幻觉）
- 摘要 ROUGE-L ≥ 0.45（与人工参考对比）
- 每条摘要至少包含 3 个有效引用

### 3.2 P0 · 数据采集

#### 3.2.1 新闻源接入

**描述**：建立多渠道新闻数据采集体系。

**数据源（MVP 阶段）**：

| 类型 | 数量 | 示例 |
|---|---|---|
| 主流媒体 RSS | 20+ | 新华社、人民日报、Reuters、AP |
| 新闻 API | 5+ | NewsAPI、GDELT、媒体开放 API |
| 行业垂直站 | 10+ | 36氪、虎嗅、TechCrunch |

**功能点**：

- 定时抓取，频率可配置（默认 15 分钟）
- 数据清洗：去广告、去导航、提取正文
- 数据标准化：统一字段格式（标题、正文、来源、时间、作者）
- 增量更新，避免重复入库

**验收标准**：

- 数据采集延迟 ≤ 30 分钟（从发布到可搜索）
- 正文提取准确率 ≥ 95%
- 系统 7×24 稳定运行，采集成功率 ≥ 99%

### 3.3 P1 · 事件追踪

#### 3.3.1 事件聚合

**描述**：自动将多篇关于同一事件的报道聚合为一个事件簇。

**功能点**：

- 基于语义相似度自动识别同一事件的不同报道
- 为每个事件簇生成事件标题和摘要
- 事件时间线：按时间顺序展示事件发展脉络
- 标注事件中的关键实体（人物、机构、地点）
- 展示事件的情感走向变化

**验收标准**：

- 事件聚合准确率 ≥ 85%（Precision）
- 事件聚合召回率 ≥ 80%（Recall）
- 同一事件下报道按时间排列无误

#### 3.3.2 事件订阅

**描述**：用户可订阅感兴趣的事件，有新进展时收到推送通知。

**功能点**：

- 一键订阅某个事件
- 事件有新报道时，判断是否为"实质性进展"（而非重复报道）
- 通过 App 推送 / 邮件 / 微信通知用户
- 推送内容包含简短摘要，用户可点击查看详情

**验收标准**：

- 新进展推送延迟 ≤ 1 小时
- 推送精准率 ≥ 90%（不推送重复/无实质进展的内容）

### 3.4 P1 · 个性化与订阅

#### 3.4.1 关键词/话题订阅

**功能点**：

- 用户设置关键词、话题、行业进行订阅
- 支持邮件日报/周报和实时 App 推送两种模式
- 可设置推送频率和静默时段

#### 3.4.2 个性化推荐

**功能点**：

- 基于阅读历史构建用户兴趣画像
- 搜索结果排序融入个性化因子
- 首页提供个性化新闻 Feed
- 用户可手动调整兴趣偏好

### 3.5 P1 · 多维度筛选

**功能点**：

- 时间范围筛选：今天 / 本周 / 本月 / 自定义
- 来源媒体筛选：按媒体名称或媒体类型
- 地域筛选：国内 / 国际 / 指定地区
- 语言筛选：中文 / 英文 / 多语言
- 情感倾向筛选：正面 / 中性 / 负面
- 高级搜索：支持布尔运算（AND / OR / NOT）

### 3.6 P2 · 深度能力

#### 3.6.1 多轮对话式搜索

- 支持上下文关联的追问，如"那欧洲呢？""更详细地说说第二点"
- 对话历史可保存和回溯

#### 3.6.2 跨语言搜索

- 中文 query 可检索英文新闻，反之亦然
- 结果自动翻译为用户首选语言

#### 3.6.3 行业垂直定制

- 针对金融、科技、医疗等行业提供专属搜索模式
- 行业术语识别和专业知识图谱

## 4. 搜索效果评估体系

### 4.1 评估框架总览

```
                    ┌────────────────┐
                    │   评估体系      │
                    └───────┬────────┘
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
     ┌──────────┐   ┌──────────┐   ┌──────────┐
     │ 离线评估  │   │ 在线评估  │   │ 人工评估  │
     │(上线前)   │   │(上线后)   │   │(定期)     │
     └──────────┘   └──────────┘   └──────────┘
```

### 4.2 离线评估

**目的**：在上线前验证搜索排序和 AI 摘要的基础质量。

#### 4.2.1 评估数据集

- 规模：500+ 标注 query-document 对
- 构成：
  - 热门 query（40%）：高频搜索词
  - 长尾 query（30%）：低频但有明确意图
  - 自然语言 query（30%）：完整问句
- 标注方式：3 人标注取多数，相关性分 4 级（完美匹配 / 高度相关 / 部分相关 / 不相关）
- 更新频率：每月新增 50+ 标注对

#### 4.2.2 搜索质量指标

| 指标 | 定义 | 目标值 | 计算方式 |
|---|---|---|---|
| NDCG@10 | 前 10 条结果的归一化折损累积增益 | ≥ 0.75 | 考虑位置折损的排序质量 |
| MAP | 平均精确率 | ≥ 0.70 | 所有相关文档的平均精确率 |
| Recall@50 | 前 50 条结果中相关文档的召回率 | ≥ 0.85 | 相关文档被检索到的比例 |
| MRR | 平均倒数排名 | ≥ 0.60 | 第一个相关结果的平均位置 |
| P@5 | 前 5 条结果的精确率 | ≥ 0.70 | 前 5 条中相关的比例 |

#### 4.2.3 AI 摘要质量指标

| 指标 | 定义 | 目标值 |
|---|---|---|
| ROUGE-L | 摘要与参考文本的最长公共子序列重叠度 | ≥ 0.45 |
| 事实准确率 | 摘要中无幻觉/错误陈述的比例 | ≥ 95% |
| 引用准确率 | 摘要中引用与原文对应的比例 | ≥ 98% |
| 完整性评分 | 摘要覆盖核心信息点的比例（人工标注） | ≥ 80% |

#### 4.2.4 执行机制

- 每次模型/策略更新前，自动跑离线评估 Pipeline
- 结果自动生成评估报告，指标低于阈值则阻断上线
- 评估报告归档，支持历史趋势对比

### 4.3 在线评估

**目的**：在真实流量中持续监控搜索效果。

#### 4.3.1 核心指标

| 指标 | 定义 | 目标值 | 采集方式 |
|---|---|---|---|
| 搜索成功率 | 搜索后产生了点击行为的 session 占比 | ≥ 70% | 前端埋点 |
| 零结果率 | 搜索无结果的 query 占比 | ≤ 5% | 服务端日志 |
| 首次点击位置（MRR） | 用户首次点击的平均排名位置 | MRR ≥ 0.6 | 前端埋点 |
| 二次搜索率 | 同一 session 中换词重搜的比例 | ≤ 20% | 前端埋点 |
| AI 摘要采纳率 | 用户阅读摘要后不再点击原文的比例 | 仅监控 | 前端埋点 |
| 平均停留时长 | 结果页平均阅读时间 | 合理区间 | 前端埋点 |
| 搜索响应时间 | 从发起搜索到结果返回的 P95 延迟 | ≤ 3s | 服务端监控 |

#### 4.3.2 用户满意度指标

| 指标 | 定义 | 目标值 | 采集方式 |
|---|---|---|---|
| 结果页满意度 | 搜索结果页底部的"有帮助"点击率 | ≥ 60% | 前端交互 |
| NPS | 净推荐值（周期性调查） | ≥ 40 | 问卷调查 |
| CSAT | 客户满意度评分 | ≥ 4.0/5.0 | 问卷调查 |

#### 4.3.3 监控看板

- 搭建 Grafana 实时监控看板，展示上述核心指标
- 配置告警规则：指标异常波动时自动告警
- 按日/周/月生成评估报告

### 4.4 人工评估

**目的**：弥补自动化指标的盲区，深入分析具体案例。

#### 4.4.1 Side-by-Side 对比评估

- **频率**：每两周一次
- **方法**：抽取 100 个 query，同时在本产品和竞品（Google News / 百度资讯）中搜索
- **评判维度**：结果相关性、信息全面性、时效性、来源多样性
- **评判者**：3 位评估员独立打分，取平均
- **目标**：胜出率（本产品优于竞品的比例）≥ 60%

#### 4.4.2 Bad Case 审查

- **频率**：每周一次
- **方法**：从在线日志中提取低点击率 / 高二次搜索率的 query 各 50 个
- **分析维度**：
  - 召回问题：数据源未覆盖？
  - 排序问题：相关结果排名靠后？
  - 摘要问题：AI 摘要不准确或不完整？
  - Query 理解问题：意图识别错误？
- **输出**：Bad Case 分析报告 + 改进 Action Item

#### 4.4.3 AI 摘要幻觉检测

- **频率**：每周一次
- **方法**：随机抽取 100 条 AI 摘要，人工核实与原文的一致性
- **标注**：标记幻觉类型（事实错误 / 来源张冠李戴 / 过度推断）
- **目标**：幻觉率 ≤ 5%

### 4.5 A/B 测试框架（P2）

- 支持按用户/流量百分比分流
- 支持同时运行多组实验
- 实验效果自动统计，达到统计显著性后通知
- 典型实验场景：排序策略对比、摘要模型对比、UI 交互方案对比

## 5. 技术方案

### 5.1 技术选型

| 组件 | 推荐方案 | 备选方案 | 选型理由 |
|---|---|---|---|
| 搜索引擎（关键词） | Elasticsearch 8.x | OpenSearch | 生态成熟，BM25 支持完善 |
| 向量数据库 | Qdrant | Milvus / Weaviate | 轻量高效，支持混合检索 |
| Embedding 模型 | BGE-M3 | GTE-large / E5 | 多语言支持好，效果领先 |
| LLM（摘要/对话） | Claude API | GPT-4o / 自部署开源 | 长上下文、低幻觉率 |
| Re-Ranking 模型 | BGE-Reranker-v2 | Cohere Rerank | 可自部署，成本可控 |
| 数据采集 | Scrapy + RSSHub | Crawlee | Python 生态，社区活跃 |
| 后端框架 | FastAPI (Python) | Go + Gin | 与 ML 生态无缝集成 |
| 前端框架 | Next.js | Nuxt.js | SSR 支持，搜索产品 SEO 友好 |
| 消息队列 | Redis Streams | Kafka | MVP 阶段轻量够用 |
| 监控 | Grafana + Prometheus | DataDog | 开源免费，可视化强 |

### 5.2 关键流程

#### 搜索请求处理流程

```
1. 用户输入 query
2. Query 预处理（分词、纠错、敏感词过滤）
3. 意图识别（分类：事实查询 / 事件追踪 / 观点搜集 / 趋势分析）
4. Query 改写（同义扩展、时间范围推断、实体链接）
5. 并行召回
   ├── BM25 关键词检索 → Top 100
   └── 向量语义检索 → Top 100
6. 结果融合（RRF / 加权合并）→ Top 200
7. AI Re-Ranking → Top 50
8. 去重 & 多样性调整 → Top 20
9. AI 摘要生成（基于 Top 10 文档）
10. 返回结果（摘要 + 排序列表 + 筛选面板）
```

### 5.3 成本控制策略

| 策略 | 说明 | 预估节省 |
|---|---|---|
| 热门 Query 摘要缓存 | 相同/相似 query 在时效性窗口内复用摘要 | LLM 调用 -40% |
| 分级摘要生成 | 简单 query 用小模型，复杂 query 用大模型 | LLM 成本 -30% |
| Embedding 批量计算 | 新闻入库时批量向量化，非实时计算 | GPU 资源 -50% |
| 冷数据归档 | 超过 90 天的新闻迁移到低成本存储 | 存储成本 -60% |

## 6. 产品路线图

### 6.1 分期规划

```
Phase 0 · MVP（第1-6周）
├── 接入 20+ 核心新闻源（RSS + API）
├── 混合检索 + 基础 Re-Ranking
├── AI 摘要回答（单轮）
├── 基础离线评估 Pipeline
├── Web 端搜索界面（搜索框 + 结果列表 + 摘要卡片）
└── 基础运维监控

Phase 1 · 体验增强（第7-10周）
├── 事件聚合 & 时间线
├── 个性化排序
├── 关键词/话题订阅 & 推送
├── 多维度筛选面板
├── 在线评估埋点上线
└── Grafana 评估看板

Phase 2 · 深度能力（第11周起，持续迭代）
├── 多轮对话式搜索
├── 跨语言搜索
├── 行业垂直定制
├── A/B 测试框架
└── 移动端适配
```

### 6.2 里程碑

| 里程碑 | 时间 | 交付物 | 成功标准 |
|---|---|---|---|
| M1：技术验证 | 第 2 周末 | 混合检索 Demo | 50 个测试 query，NDCG@10 ≥ 0.65 |

## 7. 数据需求

### 7.1 数据采集

| 数据类型 | 来源 | 采集频率 | 存储量预估（月） |
|---|---|---|---|
| 新闻正文 | RSS / API / 爬虫 | 每 15 分钟 | ~500 万篇 |

### 7.2 数据治理

- 数据脱敏：用户行为数据做匿名化处理
- 版权合规：仅存储摘要和元数据，不全文转载受版权保护的内容
- 数据保留：原始数据保留 12 个月，聚合数据永久保留
- 隐私合规：符合 GDPR / 《个人信息保护法》要求

## 8. 风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|---|---|---|---|
| AI 摘要产生幻觉 | 用户信任度下降 | 中 | 强制引用溯源 + 每周幻觉审查 + 事实性检查模型 |
| LLM API 成本超预期 | 运营成本不可控 | 中 | 热门缓存 + 分级调用 + 预算告警 |
| 新闻源被封禁/限流 | 数据采集中断 | 高 | 多源冗余 + 优先官方 API + 合规爬取 |
| 搜索延迟过高 | 用户体验差 | 低 | 异步摘要 + 先出列表后出摘要 + 缓存预热 |
| 版权纠纷 | 法律风险 | 中 | 仅展示摘要+链接，不全文转载 + 法务审核 |
| 竞品快速跟进 | 差异化被削弱 | 中 | 深耕事件追踪和行业垂直场景，建立数据壁垒 |

## 10. 附录

### 10.1 术语表

| 术语 | 解释 |
|---|---|
| BM25 | 经典的关键词检索算法，基于词频和逆文档频率 |
| Embedding | 将文本转换为高维向量表示，用于语义相似度计算 |
| NDCG | 归一化折损累积增益，衡量排序质量的标准指标 |
| MAP | 平均精确率，衡量检索系统整体精确度 |
| MRR | 平均倒数排名，衡量第一个相关结果的排名位置 |
| ROUGE-L | 自动摘要评估指标，基于最长公共子序列 |
| RRF | 倒数排名融合，用于合并多路召回结果 |
| Re-Ranking | 对初步召回结果进行精排，提升排序质量 |
| Query Rewrite | 对用户原始查询进行改写和扩展以提升检索效果 |
| NPS | 净推荐值，衡量用户推荐意愿的指标（-100 到 100） |
